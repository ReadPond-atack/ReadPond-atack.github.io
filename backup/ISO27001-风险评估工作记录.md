&emsp;&emsp;综述：在ISO 27001中，核心是风险评估，但是GB/T 20984-2022《信息安全技术 信息安全风险评估方法》也是风险评估。问题来了，两者都涉及风险评估，区别是什么？

### 一、风险评估在 ISO 27001 中的位置

&emsp;&emsp;ISO/IEC 27001条款6.1"Actions to address risks and opportunities"`中明确要求：组织必须通过风险评估 + 风险处置来管理信息安全风险`。其中：在6.1.2中的`Information security risk assessment`明确指出`要求组织定义并实施信息风险安全评估过程，该过程必须包括：风险识别、风险分析、风险评价`。
&emsp;&emsp;ISO27001把信息安全管理体系看成基于风险的管理体系，所以明确有哪些资产、这些资产面对哪些威胁、这些资产中哪些有脆弱性并且这些脆弱点会造成多大的影响。在根据风险大小决定要上哪些控制措施（技术、管理、物理）。这些控制措施被写入适用性声明（Statement of Applicability, SoA），而 SoA 本身就是"风险评估与处置结果与实际落地实施之间的桥梁"。

### 二、ISO 27001 对"安全风险评估"的具体要求

- 条款 6.1.2 的核心要求(这是必须要满足的)
&emsp;- 定义并维护一个正式的“风险评估方法论”
&emsp;&emsp;- 明确如何做风险识别、分析、评价；
&emsp;&emsp;- 定义影响等级和可能性等级（例如 1–5 级）及其含义；
&emsp;&emsp;- 确保多次评估结果一致、可比、可复现。
&emsp;- 确定风险可接受准则（risk acceptance criteria）
&emsp;&emsp;- 明确什么风险是“可接受 / 不可接受”；
&emsp;&emsp;- 通常由管理层（甚至董事会）批准风险偏好/容忍度
&emsp;- 风险评估过程必须包含三个子步骤
&emsp;&emsp;- 风险识别：找出资产、威胁、脆弱性；
&emsp;&emsp;- 风险分析：评估影响和可能性；
&emsp;&emsp;- 风险评价：将风险与可接受准则比较，判断是否可接受。
&emsp;&emsp;从流程上看，ISO 27001 只规定了一个高层级的过程模型："识别 – 分析 – 评价"，并不限定具体资产分类、威胁/脆弱性赋值方式、计算公式等。

### 三、ISO27001中风险评估与风险处置和SoA的关系

&emsp;&emsp;在6.1.3 Information security risk treatment中要求组织对不可接受的风险选择适当的风险处置方式（规避、降低、转移、接受），并选择相应的控制措施。风险处置的主要产出是：风险处置计划（Risk Treatment Plan）和适用性声明（SoA）。其中SoA列出决定采用的所有控制措施（通常来自 ISO 27001 附录 A）；说明哪些控制适用 / 不适用，以及适用/不适用原因以及记录每个控制的实施状态（已实施 / 计划中 / 部分实施等）。

### 四、GB/T 20984‑2022 的风险评估流程

- 流程阶段: 评估准备 → 风险识别 → 风险分析 → 风险评价
- 风险识别阶段细化为：
&emsp;- 资产识别（业务资产、系统资产、系统组件和单元资产）
&emsp;- 威胁识别（来源、主体、种类、动机、时机、频率等）
&emsp;- 已有安全措施识别（预防性 / 保护性）
&emsp;- 脆弱性识别（管理 + 技术，分别赋值“被利用难易程度”和“影响程度”）
- 风险分析：基于风险计算模型，对单个资产风险值进行计算与等级划分，再由资产风险推断业务风险
- 风险评价：依据风险评价准则，对风险结果进行等级划分，并支撑业务风险管控
同时，标准强调：以业务为最高管控对象，资产按"业务资产 → 系统资产 → 组件/单元资产"分层,增加了沟通与协商机制，要求评估过程中与内外部相关方保持沟通.

### 五、二者关系

&emsp;&emsp;ISO/IEC 27001要求组织建立并应用风险评估过程，但它不规定具体怎么评估，比如如何得出概率、影响值等细节。它更偏向“控制要求”，属于管理体系要求。
&emsp;&emsp;GB/T 20984-2022则可作为或帮助组织满足ISO/IEC 27001风险评估要求的具体方法论标准。组织完全可以结合GB/T 20984-2022来设计其风险评估流程，使其既符合ISO/IEC 27001要求，又具有更高的实施规范性和一致性。

### 六、主要差异

1. 标准定位和约束力不同
&emsp;&emsp;ISO/IEC 27001:2022是一个管理体系要求标准（Requirements），用于指导组织建立、实施、维护和持续改进信息安全管理体系（ISMS）。风险评估只是其中一部分要求，是构建整个体系的基础。它规定了组织必须“定义并应用风险评估过程”，确保其结果可重复、有效、可比较，且用于风险处置决策等，但不详细规定具体方法、步骤或计算公式。评估方法由组织自主选择（可定性、定量或混合）。其目标是通过系统化风险管理，建立并保持符合要求的 ISMS，实现信息安全的持续可控，满足认证与合规要求。覆盖组织全范围的信息安全，强调PDCA 循环与持续改进。
&emsp;&emsp;GB/T 20984-2022《信息安全风险评估方法》是一项国家标准（信息安全技术系列），专门针对风险评估的基本概念、要素关系、风险分析原理、评估流程、评估方法以及在信息系统全生命周期中如何实施等内容进行详细说明。强调风险要素（资产、威胁、脆弱性等）及它们的关系，并给出风险识别、分析、评价等方法指导。是一个过程性、方法性更强的标准，具有一定的“操作指导性”，适合作为实施风险评估时的具体参考框架。针对范围是对特定信息系统、业务场景开展专项评估，也可作为 ISMS 风险评估的技术实现手段。

2. 评估流程划分不同
- ISO/IEC 27001流程
&emsp;- 环境与范围确定：理解组织环境、相关方需求，确定 ISMS 范围。
&emsp;- 风险评估：识别→分析→评价（核心三阶段）。
&emsp;- 风险处置：选择处置措施（规避、降低、转移、接受）。
&emsp;- 控制措施实施与运行：落实控制，监控有效性。
&emsp;- 绩效评价与持续改进：内审、管理评审，优化体系。
- GB/T 20984-2022 流程（明确四阶段）
&emsp;- 评估准备：确定目标、范围、团队、依据、风险准则、方案并获批。
&emsp;- 风险识别：资产识别（业务 / 系统 / 数据）+ 威胁识别（来源 / 主体 / 动机 / 频率）+ 脆弱性识别（管理 / 技术）+ 已有安全措施识别。
&emsp;- 风险分析：计算可能性（威胁频率 × 脆弱性利用难度 × 措施失效概率）与影响（资产价值 × 损失程度），得出风险值。
&emsp;- 风险评价：对照风险准则，划分风险等级（1-5 级），区分系统资产风险与业务风险。

3. 核心要素与分析维度差异
- 资产不同
&emsp;- ISO 27001：以信息资产为核心，涵盖硬件、软件、数据、人员、流程等，强调资产对信息安全目标的支撑。
&emsp;- GB/T 20984-2022：从业务驱动出发，扩展为业务资产、系统资产、数据资源、基础网络四大类，更强调资产与业务的关联及业务连续性影响。
- 威胁与脆弱性分析
&emsp;- ISO 27001：框架性要求识别威胁与脆弱性，未细化分类与分析维度。
&emsp;- GB/T 20984-2022：威胁需分析来源（环境 / 意外 / 人为）、主体、动机（恶意 / 非恶意）、时机、频率；脆弱性从管理、技术双维度分析被利用难度与影响程度；并明确已有安全措施的有效性分析。
- 适用场景与合规导向不同
&emsp;- ISO/IEC 27001：适用于建立 ISMS 并寻求认证的组织，强调管理体系的完整性、持续性与符合性，是国际通用的管理合规标准。
&emsp;- GB/T 20984-2022：适用于国内各类组织开展专项风险评估，是等保、关基、数据安全等合规要求的技术支撑，更贴合国内网络安全监管场景。

4. 实施/审计上的区别
- ISO 27001 审核侧重点
&emsp;- 关注是否有定义好的风险评估过程（包含准则、范围、风险识别、分析、评价等步骤）。
&emsp;- 关注风险评估结果是否被用于制定风险处理计划、选择控制措施并体现在SOA等文件中。
&emsp;- 不要求必须采用某一个具体的评估模型或参考标准。
- GB/T 20984-2022 侧重点
&emsp;- 更关注评估方法本身的科学性、逻辑性、完整性（如如何识别资产、如何定义威胁、脆弱性等）。
&emsp;- 适合作为落地评估过程时的流程设计、工具选择参考标准。
